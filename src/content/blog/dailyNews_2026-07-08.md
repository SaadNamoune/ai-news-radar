### [Squish: High-performance local LLM inference server with INT3 quantization](https://squish.run/blog/local-llm-fast-enough/)

<small>**Hacker News - Newest: "llm"** &nbsp;·&nbsp; 2026-07-07 20:35:56 &nbsp;·&nbsp; `inference-server` `quantization` `local-llm` `performance-optimization`</small>

![Squish: High-performance local LLM inference server with INT3 quantization](https://squish.run/assets/blog/chart-speed.png)

**Overview:** Squish is a local LLM inference server designed to outperform existing tools (e.g., Ollama) in end-to-end response time and memory efficiency, targeting sub-5-second commit message generation. **Method:** Squish optimizes prompt reuse (exact or partial matches) and employs INT3 quantization for Qwen2.5-7B-Instruct, achieving 18% faster decoding and 11% smaller model size (3.56 GB vs 4.00 GB) with comparable accuracy (arc_easy: 0.551 INT3 vs 0.541 INT4). Benchmarks control for thermal throttling and measure decode throughput, tail latency, and TTFT (time-to-first-token) across unique and reused prompts. Squish leads in all metrics, including 9.8× faster responses for repeated prompts (200 tokens) and 1.15–1.32× faster for unique prompts. **Results:** For 4,000-token prompts, Squish returns responses in 3.8 seconds vs Ollama's 37.5 seconds (9.8× speedup). Prompt reuse scales performance non-linearly, reaching 14.7× speedup at 95% overlap. INT3 quantization works for Qwen models but fails for Gemma-3 (15-point accuracy drop). **Impact:** Advances local LLM usability by addressing latency bottlenecks in agentic workflows (e.g., git automation). Open questions include broader model compatibility for INT3 and trade-offs between quantization and accuracy across architectures.

### [VisionBridge: OpenAI-compatible proxy enabling vision capabilities for text-only LLMs](https://github.com/thomasunise/visionbridge)

<small>**Hacker News - Newest: "llm"** &nbsp;·&nbsp; 2026-07-07 20:37:55 &nbsp;·&nbsp; `multimodal-llm` `proxy-server` `openai-compatible` `vision-tools`</small>

![VisionBridge: OpenAI-compatible proxy enabling vision capabilities for text-only LLMs](https://opengraph.githubassets.com/6f14857444adcb0bf38e609780b64366029bafd628a9b3bff335c9e6ed054e02/thomasunise/visionbridge)

**Overview:** VisionBridge is an OpenAI-compatible proxy that enables text-only LLMs to process images by routing vision-related tasks to a separate vision model (VLM) via tool calls. It bridges reasoning models (e.g., DeepSeek, GLM) with vision models (e.g., LLaVA, Qwen-VL) without requiring model training or weight modifications. **Method:** VisionBridge intercepts OpenAI-style chat requests, stores images, and rewrites prompts for the reasoning model. It supports tool calls like `look(image_id, question)`, `ocr(image_id)`, `scan(image_id)`, `crop_and_look(image_id, box, question)`, and `compare(image_id_a, image_id_b, question)`. Tool calls use native OpenAI function-calling when available, falling back to a prompt-JSON protocol for unsupported backends. Images are cached content-addressably, and scene descriptions are pre-generated in parallel. **Results:** The system handles concurrent tool calls, imperfect model outputs (e.g., hallucinated image IDs), and malformed JSON by feeding observations back to the reasoning model. It supports multiple reasoning+vision model pairs via environment variables (e.g., `REASONING_BASE_URL`, `VISION_BASE_URL`) and exposes a minimal API surface (`/health`, `/v1/models`, `/v1/chat/completions`, `/v1/traces/{id}`). **Impact:** Advances multimodal LLM integration by decoupling reasoning and vision tasks, enabling flexible deployment with existing OpenAI-compatible clients (e.g., OpenWebUI, LibreChat). Open questions include scalability under high concurrent image loads and robustness to adversarial tool calls.

**Overview:** GRAFT addresses mispronunciation in text-to-speech (TTS) by enabling per-word pronunciation control using short spoken samples, improving intelligibility for rare words and technical terms. **Method:** Uses a per-word conditioning mechanism where a spoken hint is encoded via the model's tokenizer and bound to the word's position in the prompt. ### [CUTLASS-Based Fused SwiGLU Kernels for LLM Acceleration](https://arxiv.org/abs/2607.02521)

<small>**cs.DC updates on arXiv.org** &nbsp;·&nbsp; 2026-07-07 05:00:00 &nbsp;·&nbsp; `LLM-acceleration` `SwiGLU` `CUTLASS` `GPU-kernels`</small>

**Overview:** This work addresses the 9-37% execution time overhead of SwiGLU MLP activations in LLMs by fusing SwiGLU into GEMM operations using CUTLASS-based SM90 kernels. The approach targets Qwen-2.5 models (0.5B-72B) on NVIDIA H100 GPUs. **Method:** Two complementary kernels are proposed: Kernel-1 overlaps Swish computation with Up-tile loading using a Pingpong warp-specialized schedule, while Kernel-2 interleaves SwiGLU with tile stores via a custom Epilogue Visitor Tree. The fusion shifts workloads from memory-bound to compute-bound. **Results:** Achieved up to 2.47x speedup over PyTorch, reaching 79.5% peak BF16 utilization. Numerically superior to cuBLAS (0 mismatches vs. 4.5-11% mismatches), and torch.compile failed to replicate the fusion (3-7x slower). **Impact:** Demonstrates the necessity of hand-crafted tile-level designs for LLM acceleration. Advances state-of-the-art in MLP fusion, with implications for broader activation function optimizations in transformer architectures.

[→ Read full article](https://arxiv.org/abs/2607.02521)

**Overview:** This paper audits four major tool-calling benchmark families (BFCL v4, τ²-Bench, LiveMCPBench, MCP-Atlas) and finds 18.5% misalignment between evaluator scores and human judgments across 496 tasks. ### [Single-Node Optimization for GLM-5 Serving in OpenClaw Workloads](https://arxiv.org/abs/2607.02518)

<small>**cs.DC updates on arXiv.org** &nbsp;·&nbsp; 2026-07-07 05:00:00 &nbsp;·&nbsp; `LLM-serving` `distributed-inference` `performance-tuning` `throughput-optimization`</small>

**Overview:** This report analyzes GLM-5 serving optimization for OpenClaw workloads dominated by long, tool-augmented prefixes (28k-30k input tokens, 500 output tokens). The study focuses on single-node optimization within a multi-model inference architecture, evaluating chunked prefill, tensor/pipeline parallelism, and concurrency tuning. **Method:** Experiments were conducted on a two-node, sixteen-GPU cluster with configurations tested for chunked-prefill-size (2048 vs. 3072), tensor parallelism (TP=4), pipeline parallelism (PP=4), and max-running-requests (16 vs. 24). The optimal configuration (3072/TP4/PP4/max24) was identified through empirical benchmarking. **Results:** The optimized configuration improved request throughput from 0.43 to 0.48 req/s, token throughput from 9029.64 to 9993.23 tok/s, reduced average TTFT from 8.98 to 6.69s, and lowered P90 latency from 40.23 to 32.64s. Serving cost per request and per token decreased by 10.4% and 9.6%, respectively. **Impact:** Demonstrates workload-specific optimization for LLM serving, highlighting that larger chunk sizes and deeper queueing do not monotonically improve performance. Provides actionable deployment recommendations for OpenClaw workloads.

**Overview:** This paper addresses the safety challenge of when long-lived language agents should write reusable memories from execution traces. It introduces GovMem, a conservative policy that estimates dependency-aware support, retrieves counterevidence, and assigns scope to decide whether to promote, reject, or review memory writes. **Method:** GovMem uses dependency-aware support estimation and counterevidence retrieval to reduce false promotions. It is evaluated on synthetic stress tests and real-trace subsets, including a 120-candidate human-labeled dataset spanning 79 traces. **Results:** In synthetic tests, GovMem reduces false promotion from 0.597 to 0.040 while preserving 0.960 recall. In real traces, false promotion drops from 0.371 to 0.032, but held-out false promotion remains 0.111 with high review burden (0.692). External coding-agent candidates were all rejected upon adjudication. **Impact:** The work highlights memory write paths as risk-controlled evidence-governance systems, emphasizing the need for broader validation and downstream harm assessment before deployment.

### [Rootless Docker: Security Model, Implementation, and Hidden Trade-Offs](https://www.kenmuse.com/blog/rootless-docker-and-its-hidden-security-trade-offs/)

<small>**Hacker News - Newest: "security"** &nbsp;·&nbsp; 2026-07-07 16:34:56 &nbsp;·&nbsp; `rootless-docker` `user-namespaces` `container-security` `CAP_SYS_ADMIN`</small>

![Rootless Docker: Security Model, Implementation, and Hidden Trade-Offs](https://www.kenmuse.com/blog/rootless-docker-and-its-hidden-security-trade-offs/images/banner_hu_abdf2cc89198719a.png)

**Overview:** Rootless Docker mitigates the risk of a privileged Docker daemon by running `dockerd` as an unprivileged user, using per-user namespaces and UID/GID mapping via `RootlessKit` and `newuidmap`/`newgidmap`. This reduces the blast radius of a container escape or daemon compromise but introduces new kernel interface exposure risks. **Method:** Rootless Docker leverages user namespaces (creatable without root) with UID/GID mapping to simulate root privileges inside containers while remaining unprivileged on the host. It uses `fuse-overlayfs`, `slirp4netns`, or `pasta` for storage/networking, and disables seccomp/AppArmor confinement (`seccomp=unconfined`, `apparmor=unconfined`) to function. **Results:** The approach reduces host-level privileges but expands attack surface to kernel interfaces (e.g., iptables, mount system) due to user namespace capabilities (`CAP_SYS_ADMIN`, `CAP_NET_ADMIN`). Historical vulnerabilities (e.g., Qualys 2025/2026 namespace bypasses) demonstrate exploitable latent flaws in kernel code paths. **Impact:** Advances container security by eliminating root daemons but highlights the need for rigorous kernel hardening and namespace isolation. Open questions remain about balancing usability and security in rootless environments.

### [Momentic migrates from PostgreSQL to ClickHouse for scalable AI-driven cache at 2M queries/day](https://www.infoq.com/news/2026/07/momentic-postgres-clickhouse/?utm_campaign=infoq_content&utm_source=infoq&utm_medium=feed&utm_term=AI%2C+ML+%26+Data+Engineering)

![Momentic migrates from PostgreSQL to ClickHouse for scalable AI-driven cache at 2M queries/day](https://res.infoq.com/news/2026/07/momentic-postgres-clickhouse/en/headerimage/momentic-postgres-clickhouse-1783452382371.jpeg)

**Overview:** Momentic rearchitected its AI-driven testing platform cache to handle 2M+ daily queries across 20B entries with ~250ms latency by migrating from PostgreSQL to ClickHouse. This addresses PostgreSQL's performance degradation under high-write, high-read workloads with lock contention at scale. 

**Method:** ClickHouse's column-oriented architecture and sparse primary indexes (vs. B-tree) enable efficient granule-based lookups. Momentic designed a composite primary key (test ID, step ID, version, branch, commit timestamp) to optimize 90% of queries (feature branches). For main branches requiring full scans, materialized views precomputed commit timestamps per test ID to jump directly to relevant data parts. The team replaced three PostgreSQL queries with a single INSERT workflow using ClickHouse's ReplacingMergeTree for asynchronous deduplication, eliminating a Redis layer.

**Results:** Post-migration, the system scaled to 20B cache entries with 2M+ daily queries at ~250ms average latency. Migration used dual-write shadow testing with diff-based correctness checks before gradual cutover.

**Impact:** Demonstrates columnar databases' superiority for high-cardinality caching workloads in AI platforms. Open questions remain about trade-offs in analytical vs. transactional workloads and long-term operational complexity.
